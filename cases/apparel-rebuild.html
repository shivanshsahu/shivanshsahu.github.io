Three Pages, Three Wins, One Rebuild

A UK workwear and apparel brand with a fragmented site stack, legacy third-party apps fighting for layout real estate, and three underperforming surfaces: homepage, collection, and Product Detail Page (PDP). A four-month coordinated rebuild with three tests sequenced to compound — because each surface changed the qualification of the traffic reaching the next. +297% revenue on the collection redesign was only possible because the homepage test had already shifted the traffic mix.

Phase 1: The Coordination Problem

Three pages were underperforming. The naive approach would have been to test each one in sequence — homepage first, read it, then collection, then PDP. Fourteen weeks minimum, compounding delays, and each test read against a moving baseline as the site changed around it.

The right approach was to sequence them as a program with overlapping workstreams: design prototyping starting before dev finished the previous build, QA running in parallel with dev on the next variant, and all three tests scheduled to land their readouts in a coordinated analysis window.

Why sequencing mattered. If I had run these tests against each other in an un-sequenced A/B/C/D, the results would have been uninterpretable — each variant changing the conversion of the next. By sequencing, I could read each test cleanly against the new baseline established by the previous one. The +297% collection revenue was only readable because the homepage test had stabilized the traffic mix before it launched.

Phase 2: The Dev Brief That Made Handoffs Clean

The largest single risk in a multi-page rebuild is the handoff. A strategist scopes a test, a designer creates something a developer can't build, the developer builds something QA can't verify, and the test launches with broken tracking. I've seen this sequence end otherwise solid programs.

Every variant on this program got a development brief before design was finished. Specification before handoff, not after.

Test test_id: apparel-rebuild-002-collection
url_targeting: */collections/*
exclude: */collections/sale-* # excluded — different template
allocation: 50/50 · A/A verified W5d3

GA4 data layer event: view_item_list
ecommerce.item_list_id: {{collection.handle}}
ecommerce.item_list_name: {{collection.title}}
ecommerce.items[].item_id: {{product.sku}}
ecommerce.items[].item_variant: {{variant.title}}
ecommerce.items[].index: {{position}} # 1-indexed, for card-position analysis
ecommerce.items[].item_category: {{collection.type}}

Primary metric metric: collection_to_atc_rate
secondary: revenue_per_visitor, add_to_cart_position_distribution
guardrail: -10% on collection_to_purchase_rate triggers 48hr review

Responsive breakpoints: 320 · 768 · 1024 · 1440
device_tested: iPhone SE, iPhone 14, Pixel 6, iPad, desktop

No ambiguity at handoff. The first working build was already close to the spec because the spec was the scope. Every subsequent tracking question — "does this event need to fire here?" — was already answered by the brief.

Phase 3: Why I Sequenced the Tests This Way

The order was not cosmetic. It was a deliberate bet on how traffic quality compounds through a funnel:

Step 1
Homepage
Improves the quality of traffic reaching collection pages
→
Step 2
Collection
Improves the quality of traffic reaching PDPs
→
Step 3
PDP
Converts already-qualified traffic at higher rates

Running them in reverse order — PDP first — would have tested the PDP against noisy, mis-qualified traffic from the unredesigned homepage and collection. The PDP redesign's real impact would have been hidden by the bad traffic reaching it.

Phase 4: The Three Tests & Results

Test 1 — Homepage Redesign

The existing homepage prioritized brand lifestyle imagery. For a workwear buyer, lifestyle imagery is decorative — product access is what they need. The redesign elevated category access, price-point visibility, and a clear "shop by use case" entry pattern.

Test 2 — Collection Page Redesign

The collection grid was the single highest-leverage surface once the homepage was delivering qualified traffic. Product cards redesigned for information density — category, core use case, price, and the quality callout that repeatedly surfaced in the review data. This was the single biggest revenue move in the program.

With homepage and collection converting upstream buyers at higher quality, the PDP needed to catch them. The redesign tightened the first fold, moved trust signals above the Add-to-Cart button, and re-sequenced the tab navigation.

What This Engagement Taught Me

  • A multi-page rebuild is one program, not three tests. Each surface changes the qualification of the traffic reaching the next. Sequencing is a structural decision, not a scheduling convenience.
  • The dev brief is the test. A brief that specifies the GA4 data layer schema before the developer starts coding is worth more than any post-launch QA report. Ambiguous briefs produce ambiguous implementations — which produce ambiguous test reads.
  • Design, dev, and QA can run in parallel if the interfaces are clean. The interface between design and dev is the handoff brief. The interface between dev and QA is the test plan. If both are specified up-front, the workstreams overlap without stepping on each other.
  • The biggest-percentage test was only possible because of the smallest-percentage test. The +297% collection revenue was only readable because the +57% homepage orders had already stabilized the traffic mix. Credit for the collection win belongs partially to the homepage test that preceded it.
  • Legacy third-party app cleanup is test work. Before the first variant shipped, I audited and removed four legacy third-party apps fighting the new template for layout real estate. That cleanup is invisible on the case study but was the prerequisite for the test variants to behave predictably.
