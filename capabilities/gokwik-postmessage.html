When the Checkout Doesn't Talk to GTM, You Listen at the Window

GoKwik — India's dominant COD checkout platform — doesn't push to the dataLayer. It fires window.postMessage events instead. For a Gen-Z streetwear brand processing thousands of orders through GoKwik, that meant the entire checkout funnel was invisible after Add-to-Cart. This is how we got it back.

The Problem

GoKwik is embedded as a checkout overlay — not a redirect. When the customer clicks "Buy Now", GoKwik's UI takes over. Standard ecommerce tracking assumes every funnel step pushes a dataLayer.push(). GoKwik doesn't. It emits window.postMessage events with its own schema — and none of the documentation tells you this clearly.

The result: funnel reports showed a wall after Add-to-Cart. Purchase events were unreliable because GoKwik reports the order back to Shopify, which then tries to fire its own GA4 event — a broken telephone that lost attribution on a meaningful percentage of conversions.

The Architecture

A single IIFE deployed via GTM as a Custom HTML tag firing on All Pages. It attaches a window.addEventListener("message"), transforms GoKwik's proprietary payload into standard GA4 ecommerce schema, and bridges the purchase event across a page navigation using sessionStorage.

The Hard Part: The Purchase Event

The sessionStorage Bridge

GoKwik's orderPlaced event fires inside the checkout overlay, before the thank-you page loads. If you push a purchase event at that moment, Shopify's redirect wipes the dataLayer. The order data gets stored in sessionStorage on the event, then read back and fired as a GA4 purchase event when the thank-you page initializes. The session clears the entry immediately after firing to prevent double-counting on refresh.

The Code

events.js — postMessage listener + sessionStorage bridge
/* ── postMessage listener ── */
window.addEventListener("message", function(evt) {
  try {
    var gk = evt.data || {};
    if (gk.analyticsName !== "shopifyAnalytics") return;

    window.dataLayer.push({ event: "gokwik_" + gk.eventName, gokwikData: gk });

    if (gk.eventName === "orderPlaced") {
      sessionStorage.setItem("gokwik_orderPlaced", JSON.stringify(gk));
    }

    if (gk.eventName === "checkoutStarted") pushGA4("begin_checkout",    gk);
    if (gk.eventName === "addressStep")     pushGA4("add_shipping_info", gk);
    if (gk.eventName === "paymentStep")     pushGA4("add_payment_info",  gk);
  } catch(e) { console.warn("GK listener err", e); }
}, false);

/* ── Thank-you page: restore purchase from sessionStorage ── */
try {
  var cached = sessionStorage.getItem("gokwik_orderPlaced");
  if (cached) {
    pushGA4("purchase", JSON.parse(cached));
    sessionStorage.removeItem("gokwik_orderPlaced"); // prevent double-fire on refresh
  }
} catch(e) { console.warn("GK restore err", e); }

Why This Matters Beyond This Client

GoKwik processes a significant portion of D2C COD orders in India. Any Indian Shopify brand using GoKwik has this same blind spot. The same pattern applies to any third-party checkout that uses postMessage rather than dataLayer (Bolt, Fast, some headless implementations). When a tool you don't control owns part of the funnel, you build at the interface boundary — window.postMessage is a public broadcast.
