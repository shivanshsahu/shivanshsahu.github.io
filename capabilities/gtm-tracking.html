Tag Management

GTM & Event Layer

Brittle Custom HTML tags replaced with deterministic templates. Duplicate events eliminated at the source. A data layer schema documented, not improvised.

Why It Matters

Most Google Tag Manager (GTM) containers I audit are accumulated sediment — every agency that has touched the site left behind Custom HTML tags, and nobody has ever removed one. The result is inflated metrics, fragile firing logic, and tags that fight each other in ways that only surface as "mysterious" reporting anomalies.

Cleaning the container is as important as adding the next test's tags.

1. Eliminating Duplicate Event Firing

The problem pattern. Legacy Custom HTML tags combined with modern Google Tag templates produce duplicate events. A single "Add to Cart" click fires twice: once through the old custom tag, once through the new templated event. Every KPI is inflated by a factor of two. An A/B test showing a 10% lift in Add-to-Cart rate is meaningless because the baseline was fabricated.

Single-Page Application (SPA) routers make this worse — the old custom tag fires on the original page load event, the new one fires on the SPA route change, and users see both counted.

The fix. - De-duplication architecture: each event gets a unique event_id (hashed from user_id + product_id + timestamp bucketed to 5 seconds) and GA4 drops duplicates at the server side - Session-scope flags: for events that should only fire once per session (like view_subscription_page), set a GTM variable in the data layer on first fire and short-circuit subsequent fires - Tag consolidation: strip out legacy Custom HTML/JS tags entirely and replace with structured Google Tag templates fed by a deterministic dataLayer.push() schema

2. Data Layer Design

The data layer is the source of truth. Every tag in the container should read from it; none should compute its own values inline.

What the schema looks like:

dataLayer.push({
  event: 'add_to_cart',
  ecommerce: {
    currency: 'USD',
    value: 29.99,
    items: [{
      item_id: 'SKU-1234',
      item_name: 'Product Title',
      item_category: 'Apparel',
      item_variant: 'Medium · Navy',
      price: 29.99,
      quantity: 1
    }]
  }
});

This is the contract the development team implements. Every page emits events in this exact shape. Every GTM tag reads these exact fields. No custom JavaScript scraping the DOM to infer values. If a field is missing, the bug is in the data layer, not in thirty different tags guessing.

3. Replacing Custom HTML with Templates

Why it matters: Custom HTML tags are the black holes of a GTM container. They can execute anything, they bypass GTM's built-in variable escaping, and they are invisible to anyone who didn't write them. The moment the original developer leaves, the tag becomes undocumented legacy that nobody will touch because "it probably does something important."

The refactor pattern: - Identify what each Custom HTML tag is actually doing (most are firing a pixel event or extracting a value from the DOM) - Replace with a templated equivalent: Google Tag, Facebook Pixel template, or a GTM Variable that reads the data layer directly - Document the replacement in the tag notes so the next auditor (probably me, in two years) knows what it replaced

A container I audited recently had 14 Custom HTML tags. Nine of them were firing duplicate pixel events. Three were extracting values that already existed in the data layer. Two were dead — inherited from an agency that left three years ago. All 14 replaced with 11 clean templates.

4. Consent-Aware Tag Firing

Every tag in the container is configured with explicit consent state requirements. Analytics tags fire when analytics_storage is granted. Advertising tags fire when ad_storage is granted. There's no "fire if consent is ambiguous" — explicit grant or nothing. This is what keeps the setup compliant while also being what makes Consent Mode v2's modeling work properly.

The Operational Standard

The GTM container is documented. Every tag has a human-readable note explaining what it does, which team owns it, and when it was last reviewed. When a new tag is proposed, it gets added to the documentation before it ships — because the container that accumulates undocumented sediment is the container that silently corrupts the next team's analysis.
