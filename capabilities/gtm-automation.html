The Problem

The client operated eight separate Shopify properties under one brand umbrella — each with its own GTM container, each needing the same GA4 ecommerce implementation. Standard setup: dataLayer variables, custom event triggers, a GA4 Config tag, and GA4 Event tags for every funnel step (view_item, add_to_cart, begin_checkout, add_payment_info, purchase).

Done once, that's an afternoon. Done eight times, it's a week — and the eighth container will quietly differ from the first in at least two places you won't notice until a test reads wrong six weeks later.

How It Works

The script takes a single JSON config file defining the full desired state of a GTM workspace — variables, triggers, tags. It diffs that against what's in the container and creates or updates accordingly.

01
Authenticate
OAuth via InstalledAppFlow. Tokens cached — subsequent runs skip the browser prompt.
02
List existing state
Fetches current variables, triggers, and tags. Builds name → entity maps for diffing.
03
UPSERT
If name exists: compare and update if different. If missing: create. Idempotent — safe to re-run.
The detail that makes it production-safe

Tag configs reference triggers by name, not ID — IDs differ between containers. The script resolves names to IDs at runtime. Built-in triggers (All Pages, DOM Ready) use their known system IDs. If a trigger doesn't exist, the script prompts rather than silently skipping.

Key Implementation Details

UPSERT, not overwriteChecks existing name before every create. Merges only declared fields on update — doesn't clobber fields managed outside the config.
Trigger name resolutionConfig uses human-readable names. Script resolves to IDs post-creation, preventing cross-container ID mismatches.
--yes flag for batch runsNon-interactive mode skips prompts — all 8 containers run in a shell loop with no human in the loop.
GA4 Config tag dependencyDetects when a GA4 Event tag references a Config tag that doesn't exist yet — creates Config first, then Event.
Read-only field sanitizationStrips accountId, containerId, workspaceId, fingerprint before API calls to prevent rejection errors.
Workspace by name or IDPass --workspace "Default Workspace" or numeric ID.

The Code

Authentication — OAuth with token caching
def auth_gtm(client_secret_path, token_path):
    creds = None
    if os.path.exists(token_path):
        creds = Credentials.from_authorized_user_file(token_path, SCOPES)
    if not creds or not creds.valid:
        flow = InstalledAppFlow.from_client_secrets_file(client_secret_path, SCOPES)
        creds = flow.run_local_server(port=0)
        with open(token_path, 'w') as f:
            f.write(creds.to_json())
    return build('tagmanager', 'v2', credentials=creds)
Core UPSERT loop — variables → triggers → tags in dependency order
# 1. Variables first
existing_vars = list_entities(svc, workspace_path, 'variables')
for var in config['variables']:
    if var['name'] in existing_vars:
        update_entity(svc, 'variables', existing_vars[var['name']], var, auto_approve=args.yes)
    else:
        create_entity(svc, workspace_path, 'variables', var)

# 2. Triggers + build name→ID map
existing_trigs = list_entities(svc, workspace_path, 'triggers')
for trig in config['triggers']:
    if trig['name'] in existing_trigs:
        update_entity(svc, 'triggers', existing_trigs[trig['name']], trig, auto_approve=args.yes)
    else:
        create_entity(svc, workspace_path, 'triggers', trig)
trig_map = {n: d['triggerId'] for n, d in list_entities(svc, workspace_path, 'triggers').items()}

# 3. Tags — resolve trigger name refs → IDs before creating
existing_tags = list_entities(svc, workspace_path, 'tags')
for tag in config['tags']:
    trigger_ids = resolve_trigger_ids(tag, trig_map, auto_approve=args.yes)
    if trigger_ids is None: continue
    tag['firingTriggerId'] = trigger_ids
    if tag['name'] in existing_tags:
        update_entity(svc, 'tags', existing_tags[tag['name']], tag, auto_approve=args.yes)
    else:
        create_entity(svc, workspace_path, 'tags', tag)
Batch shell invocation — all 8 containers
while read -r CONTAINER_ID; do

done < containers.txt

Why This Matters at Scale

When the GA4 measurement ID changes, or a new event needs adding across all properties, the config is updated once and the script runs again. Every container reflects the same source of truth. Re-running is safe because UPSERT handles existing entities without duplication — this is infrastructure-as-code applied to tag management.
