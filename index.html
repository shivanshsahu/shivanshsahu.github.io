The Operating System

How I Work: The DDBL Framework

Every engagement follows the same four-phase structure: Diagnosing, Deciding, Building, Leading. Not because it's a framework worth naming, but because skipping any phase is how testing programs produce expensive noise instead of useful signal.

1. Diagnosing

Before changing a single element, I build a complete picture of where revenue is actually leaking. The goal is to arrive at a hypothesis that's grounded in data — not one borrowed from a best-practices list.

  • Heuristic & UX Audits: I walk through the full user journey as a buyer, flagging friction points: buried navigation, ambiguous CTAs, spec information that requires three clicks to find.
  • Quantitative Analytics: GA4 funnel breakdowns, device segmentation, and exit-rate analysis to identify exactly where drop-off is concentrated — and how differently it behaves on desktop versus mobile.
  • Qualitative Voice of Customer (VoC): For one B2B LED lighting client, a visitor survey surfaced that buyers were leaving product pages to look up the difference between T8 and T12 bulb types — not because the design was wrong, but because the specs page answered the wrong questions. That single survey response reshaped a full quarter's test roadmap. VoC tells you why users are behaving the way the heatmap shows; running both simultaneously cuts interpretation time significantly.
  • Technical Tracking Audit: If the data is broken, the test is invalid. Before every new engagement, I audit the GA4/GTM setup for cross-domain attribution failures, duplicate event firing, and data layer schema gaps. I've inherited broken tracking on nearly every client I've taken on. Fixing it first is not optional.

2. Deciding

An idea is not a hypothesis. A hypothesis is a testable prediction: if I change X because data showed Y, I expect metric Z to improve. Without that structure, you're running tests that can't generate a clean learning regardless of the outcome.

  • Prioritization: I use Impact, Confidence, and Effort scoring. A homepage redesign might score high on Impact but low on Confidence and Effort — making it a lower priority than a sticky buy box test that addresses a documented friction point with low development cost and high confidence in the behavioral signal.
  • The hypothesis format: "If I [change X] because [data showed Y], then I expect [metric Z] to improve." Every test in my pipeline is written this way. If I can't complete that sentence with evidence for Y, the test doesn't get built.
  • Device-forking: I avoid "one-size-fits-all" test designs. A dropdown menu that works for a mouse user on a wide screen fails on a 6-inch vertical viewport. I segment tests by device when the behavioral data shows divergence — and in my experience, it almost always does.

3. Building

The gap between strategy and execution is where most testing programs break. A clearly-scoped hypothesis that survives handoff intact is rarer than it should be.

  • The Master Tracker: Every test lives in a centralized pipeline with strict stages: Backlog → Scoped → Designed → In Dev → QA → Live → Analyzing. Developers don't start work on an undefined scope. QA doesn't approve tests without a tracking parity check. The tracker makes bottlenecks visible — if design takes two days and QA takes fourteen, that shows up immediately.
  • Clean handoffs: Developers receive documentation that specifies the exact URL targeting pattern (e.g., */collections/*), the required GA4 event triggers with the full ecommerce data layer schema, the expected CSS behavior on each breakpoint, and the success metric. Not "update the button" — the exact change, with the exact measurement attached.
  • QA protocols: Nothing launches broken. Every test passes cross-browser rendering checks, mobile responsiveness verification, tracking parity confirmation in GA4 DebugView, and an A/A allocation check before going live.

4. Leading

A winning test generates revenue. A losing test generates intelligence. Both are valuable — but only if you communicate the results honestly.

  • The guardrail check: Within the first 48 hours of a live test, I monitor for catastrophic drops in primary KPIs. If a CTA color test drops Revenue Per Visitor by 18% in the first two days, I kill it immediately. Stopping the bleeding fast is what makes loss-led testing sustainable.
  • The final readout: Results get translated into business language. Not "Variant B won at 95% confidence." Instead: the sticky header test is projected to add $7,943/month on desktop and confirmed to lose on mobile — here's why and here's how we roll it out selectively. Executives need to know what to do with the result, not just what the result was.
  • The culture shift: The most durable outcome of a well-run testing program isn't any individual test result — it's the team's relationship with evidence. When a brand stops launching features based on the founder's gut feeling and starts anchoring every decision to verified customer data, the testing program has done its real job.
